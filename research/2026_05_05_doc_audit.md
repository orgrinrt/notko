# Notko documentation audit, 2026-05-05

**Date:** 2026-05-05.
**Scope:** Repo-root README.md, three sub-crate READMEs (`notko-macros`, `notko-macros-core`, `notko-build`), Rust source rustdoc and module docs across `src/`, `notko-macros/src/`, `notko-macros-core/src/`, `notko-build/src/`.
**Method:** Manual read pass against documentation-writing + writing-style + vocabulary + ai-agent-framing workspace rules.
**Repo shape:** Notko has no `mock/` directory; it is a flat Cargo workspace with three published crates plus the root crate.

## Summary

Notko's repo-root `README.md` is in good shape: stand-alone identity, no Tier 1 leakage, no banned vocabulary, the writing-style rules look respected. The three sub-crate READMEs carry the bulk of the findings: 11 em-dashes total, one workspace-internal task ID (`#99`) leaked, and a load-bearing doc-vs-source drift where `notko-macros/README.md` describes a macro called `optimize_for(hot|warm|cold)` while the actual proc-macro is `#[profile(Hot|Warm|Cold)]`. There is also one em-dash in a test module-doc comment. Total: 13 findings across 4 README files plus 1 .rs module-doc.

## Per-file findings

### README.md (repo root)

Clean. No findings.

### notko-macros/README.md

- **Doc-vs-source drift, load-bearing.** Whole document. The README presents the proc-macro as `#[optimize_for(hot)]` / `#[optimize_for(cold)]` / `#[optimize_for(warm)]` (lowercase ident). The actual exported proc-macro at `notko-macros/src/lib.rs:22` is `pub fn profile(...)`, invoked as `#[profile(Hot|Warm|Cold)]` (capitalised idents, per `src/lib.rs:1` module doc and the root README at line 152). Every example in this README is wrong against the current API. Also: the README's "## Built-in tiers" table heading and feature names use lowercase `hot` / `warm` / `cold`, while the source uses capitalised `Hot` / `Warm` / `Cold` ZST markers. Suggested fix: rewrite all `optimize_for` references to `profile`, lowercase tier names to capitalised.
- **Tier 1 leakage, line 59-60.** `(task #99)` is a workspace-internal task ID. A reader on crates.io has no way to look up `#99`. Replace with either no reference or a public link (the GitHub repo's issues page).
- Also note: line 16-20 explains the `internal` feature with a level of detail that may belong in the lib.rs module doc instead of the README, but this is a structural suggestion, not a violation.

- **Em-dash, line 71.** `environment variables — cargo sets these on build scripts`. Replace with `:` or sentence break.
- **Em-dash, line 79.** `cargo:rustc-env=NOTKO_OPTIMISERS_PATH=$OUT_DIR/notko-optimisers — the notko-macros proc-macro reads this during expansion.`. Same fix.

- **Em-dash, line 92.** `$CARGO_MANIFEST_DIR/notko-optimizers/<name>.rs — crate-local (doesn't require notko-build).`. Same fix.
- **Em-dash, line 94.** `$NOTKO_OPTIMISERS_PATH/<name>.rs — accumulated (requires notko-build in the consumer's build.rs).`. Same fix.
- **Spelling inconsistency, structural.** The README mixes `notko-optimizers` (US spelling, line 92, 94, 98, 69) and `notko-optimisers` (UK spelling, lines 5, 64, 73, 79, 81, 89). Looking at `notko-build/src/lib.rs:25` (`LOCAL_DIR = "notko-optimizers"`) and line 28 (`OUT_SUBDIR = "notko-optimisers"`), both spellings are intentional: the source dir is US-spelled and the out-dir is UK-spelled. This is genuinely confusing on first read, even when correct. Suggested fix: add an explicit note up front explaining the deliberate spelling split (US for the source filesystem location, UK for the cargo metadata propagation key) so a reader does not flag it as a typo.
- Note: at line 60 the README does say `panic_fmt = "trace invariant violated: {err:?}"` which uses lowercase `cold` (line 58 says `based_on = "cold"`). The actual `Strategy` enum likely uses capitalised variants (the lib.rs has `Strategy::Cold` per the third-party-attribute example). Confirm this maps correctly through the file-header parsing logic; the README claims lowercase `"cold"` is what the file format takes, but if the parser expects `"Cold"` this is a doc-vs-source drift.

## Cross-cutting patterns

1. **Em-dashes still present in three out of four READMEs.** The repo-root README is clean (presumably hand-cleaned in a recent pass), but the sub-crate READMEs were not swept. Eleven em-dashes total across the three sub-crate READMEs.
2. **The `#[profile]` vs `#[optimize_for]` rename has not propagated.** The proc-macro was renamed in source but `notko-macros/README.md` and `notko-build/README.md` still describe the older `optimize_for` shape with lowercase tier names. This is the biggest finding by impact: a stranger landing on `notko-macros` from crates.io would copy the README example and find nothing compiles.
3. **Workspace-internal task IDs leak in one place** (`task #99` in notko-macros/README.md:59).
4. **Spelling inconsistency between `notko-optimizers/` and `notko-optimisers/` is intentional but undocumented**, leaving the reader to infer it. A short note up front would defuse the perceived bug.
5. **Glossary-style bullets with em-dashes** (label-plus-em-dash) appear in notko-macros-core/README.md "Public API" section. Convert to a two-column table per writing-style guidance.
